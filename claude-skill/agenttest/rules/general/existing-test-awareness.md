# Rule: Existing-test awareness

Before writing a new test class, check the project for existing test
classes for this target. Match conventions; don't duplicate.

## Search for existing test classes

For target class `<TargetClass>`, search the project's test source roots:

```bash
# Maven convention:
find src/test/java -name "<TargetClass>Test.java" -o -name "<TargetClass>Tests.java" -o -name "<TargetClass>AgentGenTest.java"

# Or use Glob: src/test/java/**/<TargetClass>*Test*.java
```

### If `<TargetClass>AgentGenTest.java` already exists

This file was previously generated by AgentTest. Read it fully. Decide:

- **Add new test methods to it** if your new tests cover orthogonal
  scenarios (different OWASP payloads, different agent patterns)
- **Replace specific test methods** if you're refining a previously-failing
  test (preserve the rest of the file — don't blow it away)
- **Skip generation** if the existing file already covers the same
  invariants — refuse and tell the user

### If `<TargetClass>Test.java` or `<TargetClass>Tests.java` exists

This is human-written. Don't merge into it (would conflate
AgentTest output with human tests, making review harder). Create
`<TargetClass>AgentGenTest.java` as a separate file. The `AgentGen`
marker makes AgentTest output reviewable independently.

### If no existing test class

Create `<TargetClass>AgentGenTest.java` fresh, following the convention
established by the rest of the project.

## Match project conventions

Even when creating a fresh test class, scan **2-3 sibling test classes**
in the same package or `src/test/java/...` to learn:

| Convention | What to extract |
|---|---|
| Test framework version | JUnit 4 vs JUnit 5 — affects imports + annotations |
| Mocking library | Mockito vs MockK (Kotlin) vs JMockit |
| Assertion library | AssertJ vs Hamcrest vs JUnit native |
| Spring test annotations | `@SpringBootTest` (avoid for unit tests) vs `@DataJpaTest` vs none |
| Test data builders | If sibling tests use Lombok `@Builder` or Test Data Builders, follow suit |
| Test method naming | `should_<behavior>` vs `<method>_<state>_<outcome>` vs `test<MethodName>` |

Apply the project's convention even if it differs from
`rules/java/junit-template.md`'s default — local consistency wins. The
exception: **never use `@SpringBootTest` for AgentGenTest classes** even
if siblings use it (CLAUDE.md rule: skill is for unit tests, not
integration).

## Reading neighbors

Use the Read tool on 2-3 sibling test files. Look for:

```java
// Imports tell you the toolchain
import org.junit.jupiter.api.Test;             // JUnit 5 ✓
// vs
import org.junit.Test;                          // JUnit 4 (older project)

import static org.assertj.core.api.Assertions.*;   // AssertJ ✓
// vs
import static org.hamcrest.MatcherAssert.assertThat;  // Hamcrest

import org.mockito.junit.jupiter.MockitoExtension;   // JUnit 5 + Mockito
```

Match the imports the project uses, even if a different choice would be
"better".

## When sibling check yields nothing

If `src/test/java` is empty (e.g., the spring-ai-examples chain-workflow
target has 0 existing tests), default to the convention in
`rules/java/junit-template.md`: JUnit 5 + Mockito + AssertJ +
`@ExtendWith(MockitoExtension.class)`.
